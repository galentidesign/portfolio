## The launch-day catch: cookies on a cookieless site

The production sweep's beacon browse (26 beacons, every one 204) ended with
a spot check that found **two cookies on the wire** — `XSRF-TOKEN` and
`_portfolio_session` — on a site whose telemetry contract and colophon say
cookieless, with 202 request specs asserting exactly that, all green.

Root cause chain: inertia_rails sets `cookies['XSRF-TOKEN'] =
form_authenticity_token if protect_against_forgery?` as an `after_action`
on every controller (cookie-jar write — the telemetry sink's
`session_options[:skip]` stops only the session cookie), and generating the
token writes the session, which drags `_portfolio_session` along. The rails
template's `test.rb` disables forgery protection, so
`protect_against_forgery?` is false exactly where the specs run — **the
suite asserted a cookieless world that only existed in the test
environment.** Staging had served both cookies since M0; nothing ever read
a Set-Cookie header off a staging response until today.

Fix: forgery protection off in every environment (application.rb, loud
contract comment) — sound because no endpoint authenticates with cookies
(`/t` skips CSRF by design, demo API is read-only, `/ops` is basic-auth) —
plus `spec/requests/cookieless_spec.rb` pinning the config AND the wire
(+10 examples). Verified where the bug lived: a local production-mode boot
(`assume_ssl`, real forgery config) showed zero Set-Cookie on every path
with `/t` still 204. The meta-lesson extends gotcha-aaa's family: an
env-dependent default means test-green proves the test environment, not the
contract — launch contracts get verified on the production wire.

A verify-the-verifier moment inside it: the first production-mode curl
batch used an unquoted shell variable for a header — puma rejected every
request as an HTTP parse error (leading space in the header name) and the
"zero cookies" readings were puma error pages. The server log, not the
status codes, told the truth; re-run clean before concluding anything.

  crisp — the unfurl box closed on both URLs the same day it was opened.
