# M8 — Studies + live demo (2026-07-05)

Ninth build session. Goal: the two case-study pages become real surfaces —
Study A's decision→build→ripple layout with neutral diagram components,
Study B's extraction-driven token-translation and API-mapping tables — and
the Study B live demo ships: the Chores flow rebuilt with actual
@shopify/polaris components, served four forceable states by Rails, inside
a bundle that core routes never download.

## Session shape

- One frontier orchestrator + four parallel mid-tier build agents on
  disjoint file areas + one mid-tier e2e agent in a second wave. The
  orchestrator pre-landed the shared substrate (deps, routes, a binding
  contract doc, the shared ProseSlot primitive) so no two agents touched a
  central file; briefs pinned git read-only and mise-pinned ruby.
- **The contract doc did the coordination.** `app/frontend/studies/README.md`
  fixed the demo API shapes, the state/latency protocol, the CSS lifecycle,
  seed texture, and test hooks before any agent spawned; backend, demo, and
  e2e agents built against it independently and met in the middle with zero
  interface rework.
- **Polaris under React 19, resolved by audit rather than docs.** Context7
  was quota-blocked, so the exact pins came from the npm registry plus a
  tarball audit: @shopify/polaris 13.9.5 declares `react ^18` peers but
  ships zero `findDOMNode`, zero `defaultProps`, and `nodeRef` on every
  transition — React-19-safe in practice. An npm `overrides` block maps its
  peers to the root React; the lockfile resolution was proven in a
  scratch dir before touching the repo.
- **The CSS quarantine was designed before the demo was built.** Polaris's
  stylesheet carries global resets (`html, body { font-size: 13px; … }`,
  `h1–h6, p { margin: 0 }`); as a normal lazy CSS chunk it would survive
  client-side navigation and restyle the site. The contract mandates
  `?inline` import + one `<style data-polaris-demo-styles>` element
  injected on mount and removed on unmount, with a site-guard block
  appended (higher-specificity `html[data-skin]` re-assertions) so shell
  chrome keeps its geometry while the demo is mounted. The e2e suite
  proves the teardown.
- One wave-1 agent stalled (stream watchdog, 600s silent) mid-way through
  a fake-timers test fight. A SendMessage resume with concrete fix
  guidance recovered it — it finished all gates green. Ping-before-respawn
  held for the second milestone running; a later status ping was answered
  mid-run ("not blocked"), avoiding a wasteful inline takeover.

## Task log

- **Demo backend (agent 1)** — the repo's first ActiveRecord tables: five
  `demo_*` migrations, namespaced `Demo::` models, deterministic idempotent
  fictional seeds (2 households / 5 children / 9 chores, fixed ids,
  `upsert_all` + pk-sequence resets), and `Demo::ChoresController` serving
  the four states with a clamped latency sleep (0–2000ms, default 450) and
  a bounded loading hold (`Rails.env.test? ? 25 : 1200` ms) the client
  re-polls against. Explicit ordering everywhere keeps payloads
  byte-stable. 38 request-spec examples.
- **Study A (agent 2)** — `studies/agentic-design-ops/`:
  OrchestrationDiagram (orchestrator → agents A/B/C → review gate →
  artifact; DOM nodes linearize the story, SVG connectors decorative),
  RippleDiagram (token source → compiled targets), PatternGallery (five
  inert DS specimens with generic captions). Page arc
  decision → build → ripple → close with five labeled ProseSlots. Neutral
  labels only.
- **Study B (agent 3)** — extraction verified against both ends: CoBlend's
  Tailwind-v4 `@theme`/`:root` tokens (read-only source access, §6.5
  guardrails) and the installed polaris-tokens/Polaris type files. 23
  token-translation rows (3 clean / 9 mismatch / 11 standardized — the
  brand-hue and tenant-palette mismatches are the story), 15 API-mapping
  rows (card-list → IndexTable; dnd-kit reorder → no Polaris equivalent),
  7 WCAG-cited a11y rows. Page renders them as DS Table content with
  Badge classifications; four ProseSlots; demo-entry Card.
- **Polaris demo (agent 4)** — `studies/polaris-demo/PolarisDemo.tsx` as
  the single lazy entry: AppProvider + minimal Frame, IndexTable index
  with client-side filter/sort, create/edit forms with the divisibility
  rule as TextField errors, delete-confirm Modal, submit → Toast "Demo
  only — nothing is persisted". View state in `?view=` query params
  (popstate-synced). Steps editor uses move-up/down buttons — deliberately
  keyboard-accessible where the source app's drag reorder is not. Reduced
  motion zeroes all twelve `--p-motion-duration-*` tokens inside the demo
  root via the `useMotionPref` gate.
- **Orchestrator integration pass** — two review catches after agent 4
  returned: the Polaris EmptyState pointed at a Shopify CDN image
  (external request from a first-party site) — replaced with a local SVG
  that Vite inlines into the chunk as a data URI; and the fetched
  households never reached the create form (stub fixtures drifting from
  the seeds) — the fetch hook hoisted to the demo root so the state
  switcher drives the data layer in every view and create/edit consume
  the same payload the index renders. React 19's `set-state-in-effect`
  lint rejected the first sync attempt; the fix derives the default
  household during render instead.
- **e2e (agent 5)** — network-level isolation assertions (the M5
  technique), four-states-from-Rails, style-leak teardown, axe across
  studies + demo views × skins, keyboard + reduced-motion coverage. Two
  of its findings were corrected rather than accepted: a failure it
  called "pre-existing" was actually caused by this session (the stub
  test still asserted the M4 study stubs the wave had replaced — test
  rescoped to the remaining stubs), and it had disabled three axe
  landmark rules to accommodate Polaris Frame's nested `<main>`.
- **Frame dropped, Banner in, carve-outs deleted.** Polaris `Frame`
  exists in the demo only to host `Toast`, and Frame renders its own
  `<main>` landmark — unfixable duplicate-main violations inside a page
  that already has one. The demo now confirms submits with a dismissible
  Polaris `Banner` (same pattern, no Frame dependency), and the axe
  suite runs the full rule set: the zero-violation bar holds with no
  exceptions, Polaris included.
- **Lighthouse, first-ever capture — three real fixes fell out.** The
  §9.1 stage-5 tooling had never been built; M8 adds `lighthouse` +
  `npm run perf:lighthouse` (≥95 × 4 categories, mobile + desktop, the
  §9.1 route list). The first run failed SEO everywhere (no meta
  descriptions — added to all eight page heads) and flagged an
  uncrawlable anchor: the DS Button's disabled-anchor mode drops `href`
  (correct) but lost its link role — it now sets `role="link"`, the
  ARIA disabled-link idiom (internal fix, zero API change). Mobile perf
  then exposed a middleware ordering bug: `Rack::Deflater` appended via
  `middleware.use` sits below `ActionDispatch::Static`, so HTML
  compressed but the 327 kB entry chunk served raw —
  `insert_before ActionDispatch::Static` fixed it. A layout-level
  `modulepreload` for the Inertia page chunk (the server knows the
  component per request; a helper walks the built manifest's import
  chain) collapses the CSR waterfall by one hop on every route, and is
  proven not to leak the Polaris chunk (dynamic imports stay out of the
  chain).

## Verify

- Isolation (manifest level): `PolarisDemo-*.js` (742 kB, Polaris CSS
  inlined) referenced only by the demo page chunk, only dynamically;
  entrypoints carry zero Polaris; the empty-state SVG emits no separate
  asset.
- Isolation (network level, e2e): core routes fetch zero JS matching
  `/PolarisDemo|shopify|polaris/i` and never carry the injected style
  element; the demo route fetches the chunk and mounts the style element;
  after client-side navigation away the style element is gone and html
  font-size reads 16px.
- Four states served from Rails and observed on the wire; loading shows
  the poll loop with a persistent skeleton; error Banner retry re-hits
  the API.
- Full local suite: **673 unit · 122 request specs · 150 e2e** (30 new
  e2e incl. the isolation, four-states, style-leak, axe, keyboard, and
  reduced-motion specs), RuboCop / ESLint / stylelint / Prettier / tsc /
  token discipline / manifest drift / leak gate — all green, exit codes
  checked explicitly.
- Sanitization review (§6.5) passed on every Study surface: neutral
  diagram labels, factual extraction rows, fictional seeds, no personal
  paths or names, no source-repo references beyond the sanctioned
  product name.
- Lighthouse (local test server, gzip, median-of-3 on the borderline
  routes): desktop **100/100/100/100 on all seven routes**; mobile
  a11y/best-practices/SEO 100 everywhere, mobile perf 96–98 on five
  routes with two stragglers — `/` 92–93 and the Button doc page 94 —
  the CSR floor on simulated 4G without brotli/HTTP-2. TBT 0 ms and CLS
  0 on every route. The official capture is staging (brotli + HTTP/2,
  both favorable) after deploy; any residue is an M10 burn-down item.
  Artifacts in `tmp/lighthouse/`.
