# M9 — Telemetry + OG + résumé + colophon (2026-07-06)

Tenth build session. Goal: the site starts measuring itself and publishing
its craft — first-party cookieless telemetry (the `visits`/`events` tables,
`POST /t`, and the beacon module), instrumentation across every surface, a
basic-auth `/ops` dashboard dogfooding the DS, server-rendered OG cards
generated by ferrum, the real `/resume` and `/colophon` pages,
sitemap/robots, and the CI riders carried from M8 (postgres 18, Lighthouse
as a CI stage).

## Session shape

- One frontier orchestrator + five parallel mid-tier build agents (wave 1)
  on disjoint file sets + one mid-tier e2e agent (wave 2). The orchestrator
  pre-landed every collision surface before spawning: all M9 routes, the
  ferrum gem, `visits`/`events` migrations + bare models (schema fixed by
  contract, both DBs migrated), the layout's server-side og-tag hook, a
  site footer (the one shared-chrome piece two agents would otherwise have
  fought over), a minimal `track.ts` skeleton so the résumé agent could
  import the beacon before the beacon agent finished it, and the binding
  contract doc.
- **The contract doc did the coordination again** (M8's pattern):
  `app/frontend/telemetry/README.md` fixes tables, the day-key derivation,
  the `/t` protocol, all ten event kinds with payload shapes and fire
  rules, the beacon API, and the /ops query shapes — backend, beacon, and
  dashboard agents built against it without seeing each other.
- **Mid-flight gate rescope.** The initial briefs told every agent to run
  the full test suites as exit gates — wrong call with five agents editing
  concurrently: each full run would see neighbors' in-flight edits as
  failures (M8's "pre-existing failure" lesson, inverted). A correction
  went to all five within minutes of spawn: narrow gates to your own
  files, ignore failures outside your ownership list, the orchestrator
  owns the integration run.
- Server-side og-tag emission is a load-bearing detail: per-page meta here
  is client-side Inertia `<Head>`, and unfurl crawlers don't execute JS —
  §8's "Rails layout emits per-page og tags" is why LinkedIn previews work
  at all. The layout hook keys off the Inertia page component captured in
  `InertiaController`, with URLs built from `request.base_url` so staging
  and the launch domain need no config split.
- Privacy is enforced by construction, not policy: the day key is
  `SHA256(secret_key_base : date : IP : UA)` truncated — computed and
  discarded in the same request, so raw IP/UA can never reach a column;
  no cookies anywhere on the wire (asserted in specs); automated headless
  UAs (HeadlessChrome, Lighthouse) class as `bot` so generation and audit
  traffic never pollutes /ops. (Playwright device profiles carry a real
  Chrome UA and count as desktop — they only ever touch local/CI
  databases, so the staging dashboard stays human.)
